# 血量安全系统

## 设计动机

在高度模组化的 Minecraft 环境中，玩家的血量可能被多种来源干扰：

- SynchedEntityData 的 `DATA_HEALTH_ID` 字段被直接写入
- `setHealth()` 被其他 mod 覆写，拒绝/修改血量变更
- `maxHealth` 被 Attribute 操作降低至危险值
- 死亡事件被取消后血量异常

本系统通过 Mixin 注入 + Capability 备份构建了多层防御体系。

---

## 真实血量 (True Health)

### 核心思路

玩家的真实血量不应从 `SynchedEntityData` 读取 — 这个 DataItem 可能已被其他 mod 的 `catchSetTrueHealth` 等机制篡改。

### 实现

```
TrueHealthMixin 注入点:
├── getHealth HEAD → 检查 Capability 备份，不一致则修复
├── setHealth RETURN → 同步写入 Capability 备份
├── isDeadOrDying HEAD → 备份血量 > 0 时强制返回 false（杜绝"假死"）
├── die HEAD → 备份血量 > 0 时取消死亡
└── tick TAIL → 每 tick 兜底检查备份血量与 DataItem 一致性
```

### 数据流

```
setHealth(20) → DataItem = 20 → 同步 → Capability backup = 20
外挂 mod 篡改 DataItem → DataItem = 1
getHealth() → 发现 DataItem(1) ≠ backup(20) → 自动修复 DataItem = 20 → 返回 20
```

### 关闭时重置

`true_health` 能力关闭时，备份血量自动重置为 0，防止备份冻结在旧值（关闭期间玩家受伤后备份未更新，重新激活时误判为"非法降血"导致血量恢复）。

## 拒绝篡改 (Reject Health Manip)

### HURT_DEPTH 标记系统

```java
RejectHealthManipMixin:
├── hurt() HEAD → 设置 HURT_DEPTH 标记 = true（表示当前在合法伤害路径中）
├── hurt() RETURN → 清除 HURT_DEPTH 标记
├── setHealth() HEAD → HURT_DEPTH = false 时拦截（来源非 hurt 路径）
└── getAttribute(MAX_HEALTH) RETURN → 注册 AttributeInstance 所有者
```

### Attribute 反篡改

```
RejectHealthManipAttributeMixin:
├── AttributeInstance.setBaseValue() HEAD
├── maxHealth < 6 时 → 遍历 ATTR_OWNER 反查所有者
└── 所有者为在线玩家 → 取消修改
```

### 共享状态

`RejectHealthManipUtil` 维护 `AttributeInstance → LivingEntity` 映射，供 AttributeMixin 查询所有者。

## 死亡抗拒 (Death Defy)

### 效果

- 冷却制：触发后进入冷却（默认 5 分钟），冷却中不会再次触发
- 触发时：取消死亡 → 清除负面效果 + 灭火 → 回满血（固定 20，不使用 getMaxHealth，因为亚波伦结界等 mod 可能临时降 maxHealth 为 1）
- 无敌期：触发后进入无敌（默认 60 秒），`DeathDefyMixin` 双层拦截（setHealth HEAD + tick TAIL 字段直写）
- HUD 显示：无敌期间渲染金色边框 + 倒计时

### 双层拦截

```
DeathDefyMixin:
├── setHealth HEAD → 无敌期间拒绝 setHealth（防止外界扣血）
└── tick TAIL → 每 tick 末用反射直写强制恢复血量（反制 DataItem 篡改）
```

### 配置

| 参数 | 默认值 | 说明 |
|------|--------|------|
| 无敌持续时间 | 1200 tick (60s) | |
| 冷却时间 | 6000 tick (5min) | |
| 伤害抗性比例 | 0.25 (25%) | 全伤害减免 |
