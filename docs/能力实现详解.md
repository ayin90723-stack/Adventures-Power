## 总体架构

### 三层实现模型

26 种能力按对抗强度分布在三个实现层，**优先使用高层，仅在高层无法覆盖时才下沉**：

| 层 | 手段 | 特点 | 典型能力 |
|----|------|------|----------|
| **事件层** | Forge 事件（`LivingHurtEvent`、`PlayerTickEvent` 等） | 兼容性最好，其他模组可感知 | 灵巧、大地之力、伤害抗性 |
| **Mixin 层** | 字节码注入原版方法（`hurt`、`setHealth`、`canBeAffected` 等） | 拦截事件系统覆盖不到的路径 | 净魂、不朽装备、死亡抗拒 |
| **反射直写层** | `HealthUtil` 直接读写 `SynchedEntityData` 内部字段、实体追踪结构 | 对抗其他模组的 Mixin/ASM 拦截，"最终手段" | 影杀、真实血量、破敌之眼兜底 |

设计约定是**事件驱动优先**：能力的业务逻辑尽量放在事件层 Handler 中，Mixin 只做最低层拦截，反射直写只作为被外部模组拦截后的兜底补刀。

### 生效门禁模型

每个能力的效果代码执行前都要通过统一的门禁链：

```
服务端检查（isClientSide 直接 return）
  → isAdventurer() 或 isFullyUnlocked()   佩戴冒险饰品激活的身份
  → isAbilityEnabled("<id>")              玩家可在能力面板（P 键）自由开关
  → （部分能力）isFullyUnlocked()          觉醒专属门禁
```

> **v1.2.7 重构**：门禁链统一封装到 `AbilityGate.isAbilityActive(player, id)`，需要 progress 对象时用 `getActiveProgress(player, id)`（避免 `isAbilityActive` + 再查一次的双重查询）。各 Handler 不再重复写 `getAdventureProgress -> isAdventurer||isFullyUnlocked -> isAbilityEnabled` 三连。百分比觉醒倍率统一走 `AbilityGate.awakenedRatio`（value/100，cap 0.95）/ `awakenedPercent`（不除 100，cap 参数）。

**门禁与觉醒的关系（v1.2.1 修正）**：所有能力**统一遵循里程碑制**——所属里程碑解锁后即可使用，`isAbilityEnabled(id)` 内部会做里程碑归属校验。以下 5 个能力属于里程碑 8~10，此前版本曾遗留 `isFullyUnlocked()` 硬门禁（未觉醒即使已解锁也不生效），v1.2.1 已统一改为 `isAdventurer() || isFullyUnlocked()` 组合门禁：

- 受击坚韧 `resilience`（里程碑 8 解锁）
- 死亡抗拒 `death_defy`（里程碑 9 解锁）
- 真实血量 `true_health` / 拒绝篡改 `reject_manip` / 旅者之力 `active_skill`（里程碑 10 解锁）

觉醒（`isFullyUnlocked()`）不再是这 5 个能力的**准入门禁**，仅作为数值/机制强化层——例如受击坚韧觉醒 +6 层上限、死亡抗拒觉醒时联动免费审判、拒绝篡改觉醒 30% 反弹等，都在各 Handler 现有门禁通过后追加。

**v1.2.5 门禁两处加固**：
- **`adventurer` 不变式**：`activateFullyUnlocked()` 现连带置 `adventurer = true`（全解锁蕴含冒险者）；`checkAndActivateAdventurer()` 检测到 `fullyUnlocked = true` 但 `adventurer = false`（版本飞升 / 旧存档数据不一致）时自动补激活。由此 `isFullyUnlocked()` 为 true 时 `isAdventurer()` 必然也为 true，组合门禁 `!isAdventurer() && !isFullyUnlocked()` 两分支互为兜底。
- **攻击方能力 `receiveCanceled = true`**：淬魂 / 影杀 / 禁疗 / 破敌觉醒等攻击方能力的事件监听改用 `@SubscribeEvent(receiveCanceled = true)`，**不再检查 `event.isCanceled()`**--当亚波伦等模组在 `LivingHurtEvent` 取消伤害时，这些能力仍正常执行。仅受伤方能力（伤害抗性）保留 `isCanceled()` 门禁。同时新增 `rawAttacker` 回溯：`source.getEntity()` 非 Player 时依次尝试 `getDirectEntity()`、投射物 owner，兼容拔刀剑 / 投射物 `causingEntity` 缺失的场景。

### 数值成长公式

有成长的能力，其数值随**已解锁里程碑数**（`count`）线性或步进增长，`countAtUnlock` 为该能力所属里程碑的序号（由 `MilestoneRegistry` 从 milestones.json 加载后动态填充）：

```
线性：value = BASE + PER_MILESTONE × (count - countAtUnlock)      // 灵巧、伤害抗性等
步进：value = BASE + STEP × ((count - countAtUnlock) / 2)         // 鸿运当头、休养生息等（每 2 里程碑一档）
固定：value = -1                                                   // 解锁即完整（灵魂绑定、净魂、不朽装备等）
```

> **v1.2.7 重构**：成长公式封装到基类--线性能力继承 `LinearGrowthAbility`（实现 `base()` / `perMilestone()` 从 `ModConfig` 取值）、步进能力继承 `StepGrowthAbility`（实现 `base()` / `step()`，默认 divisor=2）、无成长能力继承 `AbstractAbility` 用默认 `value() = -1`。`countAtUnlock` 字段与 setter 由 `AbstractAbility` 统一封装，子类构造传初始值（兜底，实际由 `MilestoneRegistry` 加载 JSON 后覆盖）。

### 觉醒机制（第 11 阶段）

全部里程碑达成并持有「冒险的终点」后 `isFullyUnlocked()` 为 true，各 Handler 在门禁后追加觉醒分支：

- **11 个数值强化能力**：数值 × `AWAKEN_MULTIPLIER`（默认 1.3），部分带硬上限钳制（如闪避/减伤 ≤ 95%）
- **14 个机制质变能力**：新增行为（三段跳、AOE 斩杀、免费审判、过量吸血转护盾等），使用各自独立的觉醒配置项

## 共享基础设施

多个能力共用以下工具类，后文不再重复展开。

### HealthUtil — 血量反射工具（`util/HealthUtil.java`)

对抗外部模组篡改血量的核心武器库。原理：原版血量存储在 `SynchedEntityData`（实体同步数据）中，`getHealth()`/`setHealth()` 只是访问器——外部模组常用 Mixin/ASM 拦截访问器，但很难拦截对底层数据结构的直接读写。

| 方法 | 用途 |
|------|------|
| `getHealthDirect()` | 反射拿 `DATA_HEALTH_ID`（SRG `f_20961_`）直接从 `SynchedEntityData` 读真实血量，绕过 `getHealth()` 覆写 |
| `setHealthDirect()` | 同理直接写入，绕过 `setHealth()` 覆写 |
| `setAllHealthLikeDirect()` | 写原版血量 + 通过 `scanCustomHealthKeys()` 扫描实体类继承链上所有 `EntityDataAccessor<Float>` 字段（带缓存），把自定义血量条目一并写入--对付把血存在自定义字段里的 Boss。**v1.2.5 两处增强**：① **参照血量选择**--`getHealth()` 与 `getHealthDirect()` 差 >1.0 时判定原版血条被架空（亚波伦等 Boss 重写 `getHealth()` 返回真实血量），改用 `getHealth()` 作参照，否则扫不到差距过大的自定义血量；② **值范围约束**--Float 值需落在参照血量 10% 容差内（或差 ≤1.0）才认定为自定义血量 key，排除史维特 `DATA_WATER_DAMAGE_SCALE` 等非血量 Float 误写（曾导致 `onSyncedDataUpdated -> Slime.move` 卡死） |
| `setAllHealthLikeRaw()` | 三步饱和写入：快照当前血量 → `setAllHealthLikeDirect` → 遍历 `SynchedEntityData.itemsById` 的所有 `DataItem`，对值≈快照值的条目**直接写 `DataItem.value` 字段**（支持 Float/Double/Integer），完全不经过 `data.set()`，绕过 dirty 标记、监听器和一切 Mixin |
| `clearNegativeFloatDeltas()` | 把 DataItem 中所有负值 Float 归零。反制某些 Boss 向玩家注入"负血量偏移量"再 ASM 篡改 `getHealth()` 叠加的手法。**仅供 TrueHealth 防御侧调用，攻击侧不用** |
| `restoreFoodData()` | 反射直写 `FoodData.foodLevel`（`f_38696_`）= 20、`saturationLevel`（`f_38697_`）= 20，绕过 `setFoodLevel()` 拦截 |
| `removeDirect()` / `setRemovedDirect()` / `setRemovedFieldDirect()` | 三种强度的实体移除：反射调 `remove()` → 反射调 private `setRemoved()`（不参与虚分派）→ 直接写 `Entity.removalReason` 字段（`f_146795_`） |
| `clearRemovedFlag()` | 写 `null` 到 `removalReason`，把被外部标记移除的玩家"复活" |
| `eradicateFromWorld()` | 从 Minecraft 内部 6 个实体追踪结构直接抹除实体（详见[影杀](#影杀-shadow_kill)） |

### InvulClearUtil — 自定义无敌清除

`clearCustomInvulTimers(LivingEntity)` 遍历实体类继承链上的所有字段，按命名模式反射清除模组 Boss 的自定义无敌：

- `int` 字段名含 `invul`/`Invul` -> 置 0（如 Goety 的 `moddedInvul` / `obsidianInvul`）
- `boolean` 字段名含 `invul`/`Invul` -> 置 false
- `boolean` 字段名含 `vulnerable`/`Vulnerable` 且不含 `invul` -> 置 true（如 Mowzie's Mobs 钢铁守护者的 `vulnerable`）

> Java `String.contains()` 大小写敏感，故对每个模式显式覆盖大写开头变体（`Invul` / `Vulnerable`）。

### FriendlyFireProtection — 友伤保护

`isOwnerTarget(attacker, target)`：目标是 `TamableAnimal` 且主人是攻击者本人时返回 true。所有"超模"进攻能力（淬魂、影杀、破敌之眼、审判、嗜血）在生效前都先过这个检查，保证不会误伤自己的狼、猫或其他模组的驯服生物。

### FortuneContext — 跨 Mixin 上下文传递

两个 `ThreadLocal`（`CURRENT_BREAKER` / `AWAKENED`），解决"Mixin 注入点参数里没有玩家引用"的经典问题（详见[鸿运当头](#鸿运当头-fortune_favor)）。

### AbilityGate - 能力门禁与觉醒倍率

`isAbilityActive(player, id)` 封装"冒险者/觉醒 + 能力启用"三连门禁；`getActiveProgress(player, id)` 返回通过门禁的 `IAdventureProgress`（避免 `isAbilityActive` + 再查一次的双重查询）；`awakenedRatio` / `awakenedPercent` 统一百分比能力的觉醒倍率计算（含 cap 钳制）。各 Handler 门禁与倍率统一走此类。

### PiercingGazeUtil - 破敌之眼穿透结算

`resolveAttacker(source)` 追溯真正攻击者（直接实体 -> 间接实体 -> 弹射物 owner）；`postHurtEvent` 手动 post `LivingHurtEvent` 并取 `max(原值, 事件值)` 防 Boss 限伤；`invokeActuallyHurt` + `afterPierceFallback`（血量直写兜底 + 清自定义无敌字段）三连穿透结算，破敌之眼三层 Mixin 共用。

### DamageUtil - 内部伤害源

`isInternalSource(source)` 判断 `soul_strike` / `judgment`（防淬魂/审判的递归重入）；`createSoulStrike` / `createJudgment` 构造模组内部伤害源（绕过护甲/无敌/附魔/冷却）。

### ItemInventoryHelper - 物品栏遍历

`forEachAdventureSlot` / `forEachInventoryList` / `forEachCuriosSlot` 统一遍历玩家背包/盔甲/副手/Curios，消除 `AdventureProgressCapability` 多处遍历样板。`hasAnyAdventureSlot` 短路判断存在性。

### PersistentDataKeys - NBT key 常量

集中管理所有自定义 persistentData key（`MME_AdventureProgress` / `AP_ShadowHP_Data` 等）+ `milestoneNbtKey(id)`，避免散落字符串和跨文件重复方法，存档兼容性修改只改一处。

### AdventureItemNbtUtil - 冒险饰品 NBT

`playerHasAdventureItem/End`、`recoverProgressFromItems`（从物品 NBT 兜底恢复 Capability）、`syncAllAdventureItemNbt`、`writeMilestonesToStack`、`migrateOldStage`（旧 AdventureStage 迁移）、`replaceBeginWithEnd`（全解锁时把冒险的开始替换为终点）。物品 NBT 是数据三层备份之一。从 AdventureProgressCapability 拆出。

### SyncUtil - 数据同步

`syncCapabilityToPersistent`（Capability -> persistentData 显式同步，安全措施）+ `syncToClient`（向客户端推送含里程碑注册表元数据，供 tooltip 渲染 + 能力可用性判断）。grantMilestone / 各 handler / AdventureItemNbtUtil 共用。

### ScoreboardUtil - 计分板

`updateScoreboard` 维护"冒险解锁"计分板目标（`adventure_power_unlock`），全解锁时置 1，供命令方块/数据包查询玩家全解锁状态。

### BuffExclusionManager - Buff 排除管理

`toggleBuffExclusion` / `getBuffExclusionSet` 管理恩赐永驻黑名单（玩家在统一面板 Buff永驻标签排除不想续期的正面效果），存 persistentData 跨死亡持久。

## 里程碑 1 · 初次夜冕

### 灵巧 `agility`

**效果**：受到任何伤害时有概率完全闪避（伤害归零，不触发后续减伤链）。闪避率 10% → 73%（随里程碑成长），觉醒后 ×1.5 封顶 95%。

**实现**：
- 入口：`LivingAttackEvent` → `CombatAbilityHandler.onLivingAttack()`
- 通过门禁后计算闪避率 `(AGILITY_BASE + AGILITY_PER_MILESTONE × 成长数) / 100`，`player.getRandom().nextFloat()` 判定
- 命中闪避则 `event.setCanceled(true)` **在伤害进入减伤/护甲计算前整体取消**，并生成 20 枚 `WITCH` 粒子反馈
- 觉醒：概率 × `AWAKEN_MULTIPLIER`，硬钳制 ≤ 0.95，防止完全无敌

配置：`agility_base`(10) / `agility_per_milestone`(7) / `awaken_multiplier`(1.3)

### 大地之力 `digging_power`

**效果**：挖掘速度 ×1.5 → ×1.75，觉醒后约 ×2.625，且无视水中/空中挖掘惩罚。

**实现**：
- 入口：`PlayerEvent.BreakSpeed` → `ExplorationAbilityHandler.onBreakSpeed()`
- `event.setNewSpeed(originalSpeed × multiplier)`，纯事件层实现，无 Mixin，兼容性最好
- 觉醒：反向补偿原版 `getDestroySpeed` 的挖掘惩罚 —— 眼在水中且无水下速掘附魔时 `/5`、未着地时 `/5`，觉醒时各 `×5` 补回，叠加 `×1.5` 倍率（飞行/游泳也无惩罚）

配置：`digging_power_base`(1.3) / `digging_power_per_milestone`(0.05) / `awaken_multiplier`(1.3)

### 恩赐永驻 `perpetual_blessing`

**效果**：正面药水效果持续时间低于保底阈值时自动续期，Buff 永不自然耗尽。玩家可在统一面板（P 键 -> Buff永驻标签）把不想续的效果加入黑名单。

**实现**：
- 入口：`PlayerTickEvent`(END) → `AdventureProgressCapability.onPlayerTick()`，用 `lastBuffCheck` Map 节流为每 60 tick 检查一次
- `extendBeneficialEffects()`：遍历 `player.getActiveEffects()`，只处理 `MobEffectCategory.BENEFICIAL`；效果剩余时间 < `阈值 = BUFF_MIN_DURATION + BUFF_EXTEND_AMOUNT` 时用新 `MobEffectInstance` 覆盖（保留 amplifier/ambient/visible 参数）
- 黑名单存在 `player.getPersistentData()` 的 `MME_BuffBlacklist` 键下，跨死亡持久
- 觉醒：正面效果无法被驱散 —— 监听 `MobEffectEvent.Remove`，觉醒+恩赐永驻开启时对 `BENEFICIAL` 效果取消移除。覆盖 `removeEffect` / `removeAllEffects` / `cureEffects`(牛奶桶) 三条主动移除路径；自然过期走独立的 `MobEffectEvent.Expired` 不受影响。负面/中性照常可清

**配置项分工说明**（易混淆点）：
- `BUFF_MIN_DURATION`(400) / `BUFF_EXTEND_AMOUNT`(0) — **业务实际使用**，决定阈值和触发延长后写入的新持续时间
- `PERPETUAL_BLESSING_BASE`(400) / `PERPETUAL_BLESSING_STEP`(200) — **仅供 UI 展示**（`PerpetualBlessingAbility.value()`），面板/tooltip 里显示的"当前延长量"，业务逻辑完全不读
- 默认 `buff_extend_amount = 0` 时基础续期阈值保持默认；觉醒「正面效果无法被驱散」始终生效，不再依赖 `buff_extend_amount` 是否非零

配置：`buff_min_duration`(400) / `buff_extend_amount`(0)（觉醒「无法驱散」无配置项，固定生效）

## 里程碑 2 · 初尝败绩

### 虚空踏步 `void_step`

**效果**：空中可再次跳跃（二段跳），保留疾跑冲量、清空摔落距离。觉醒后升级为三段跳。

**实现**（客户端预测 + 服务端权威，两端共用 `VoidStepMovement` 施力公式）：
1. **客户端**（`JumpInputHandler`，监听 `ClientTickEvent`）：**边沿检测**（`jumpDown && !wasJumpDown`，按一下跳一次，松开再按再触发），**无强制冷却**；本地维护 `clientJumpsUsed`（以 `getMaxAirJumps()` 为天花板）与 `currentAirPhaseId`；条件满足时调 `VoidStepMovement.applyJump(player, power, false)` 做即时预测（`addSprintBoost = false`，**仅动 Y**，水平冲量交服务端施加），同时发送携带 `airPhaseId` 的 `DoubleJumpPacket`
2. **服务端**（`DoubleJumpHandler.handleDoubleJump(player, airPhaseId)`，网络包处理）：重新校验门禁，`canDoubleJump()` 检查不骑乘 / 不在水中 / 不在飞行 / 不在地面；计数模型从 `JUMPS_USED` 改为 `AirPhaseState`--每个空中周期绑定唯一递增 `airPhaseId`，`clientAirPhaseId < expected` 丢弃（陈旧包），`clientAirPhaseId > expected` 重置为新周期（**v1.2.3 修复"客户端跳数计数缺失导致突然飞出去"的核心**）；落地不再清理 state
3. 跳跃力：`0.42 × 方块 JumpFactor + 跳跃提升加成`，整体 × `VOID_STEP_POWER`（默认 1.2，可配置）；疾跑时沿视线方向附加 0.2 水平冲量；`fallDistance = 0`；**统一 motion 同步（防弹飞）**：无论校验是否通过，服务端**总是发送 `ClientboundSetEntityMotionPacket`**，把客户端预测拉回服务端权威值（旧版静默 return 不发包导致悬停 / 斜飞）
4. **`VoidStepMovement`**（v1.2.3 新增）：两端共用的 `calculateJumpPower` / `applyJump` / `getMaxAirJumps`；Y 速度取 `max(当前Y, power)` 消除三段跳顿挫
5. 生命周期：登出清理 `AirPhaseState`（落地不再清理）
- 觉醒：`getMaxAirJumps()` 返回 `AWAKEN_VOID_STEP_JUMPS - 1`（默认 3-1=2 次空中跳），即**三段跳**

配置：`void_step_power`(1.2) / `awaken_void_step_jumps`(3)

- **精准回收**（v1.1.6 修复）：关闭能力时必须同时满足 ①当前 mayfly=true ②`soarGrantedFlight == true` ③非创造非旁观，才回收。**装备鞘翅环、其他模组给的飞行不会被误没收**
- 飞行速度写 `flyingSpeed` 字段，只在与目标值偏差 > 0.0001 时写入——防止每 tick 重复乘法导致速度指数爆炸
- 时序处理：`Clone` 跨维度保留飞行中状态；`PlayerChangeGameModeEvent` 立即重新授予；`grantMilestone()` 解锁时立即同步（不等下一 tick，消除竞态）

配置：`awaken_soar_speed`(1.5)

### 淬魂之力 `soul_quench`

**效果**：攻击附加真实伤害：`固定值 + 目标最大生命% + 当前生命%`（9/10 里程碑两档：2+1%+1% / 4+2%+2%）。对禁疗标记中的目标 ×1.5。觉醒后目标血量 ≤20% 时整体翻倍（斩杀线）。

**实现**：
- 入口：`LivingHurtEvent` → `CombatAbilityHandler.handleSoulQuench()`
- 追加伤害用**自定义伤害类型 `adventure_power:soul_strike`**（`data/adventure_power/damage_type/soul_strike.json`，datapack 标签配置为绕过护甲/无敌/附魔/攻击冷却）走一次独立 `target.hurt()`
- 防递归：`isInternalSource()` 检查伤害源 msgId 是 `soul_strike`/`judgment` 则跳过（否则自己的追加伤害会再触发自己）
- 伤害后立即 `invulnerableTime = 0` + 反射清 `hurtTime`（部分 Boss 拿 `hurtTime>0` 当额外无敌判据）
- **直写补刀**：比较 `实际造成 < 期望 - ε`，说明 hurt 被外部拦截 → `setAllHealthLikeRaw` 直写修正后血量；归零则反射置 `deathScore = -1`（防重复计分）后强制 `target.die(source)`

### 破敌之眼 `piercing_gaze`

**效果**：攻击无视一切无敌——原版无敌帧、模组 Boss 的自定义无敌、限伤、"必须先破条件才能打"的机制统统穿透。觉醒后破无敌一击，目标 3 秒内无法获得无敌帧。

**实现**：模组内最复杂的拦截体系，5 层纵深防御分布在 4 个 Mixin + 1 个事件 Handler。核心思路：**Boss 的"无敌"实现五花八门，每层针对一类实现**。

| 层 | 位置 | 针对的无敌实现 |
|----|------|---------------|
| **Layer 0** | `PiercingGazePlayerAttackMixin`：`@Redirect` 替换 `Player.attack()`（`m_5706_`）内部的 `target.hurt()` 调用 | 完全重写 `hurt()` 且不调 `super.hurt()` 的 Boss（如暮色森林巫妖）--此时 Layer 2/2.5 的注入点根本不会执行。hurt 返回 false 时手动 post `LivingHurtEvent` 再经 Accessor 直调 `actuallyHurt()`，返回 true 保击退/附魔正常 |
| **Layer 1** | `PiercingGazeMixin`：`Entity.isInvulnerableTo()`（`m_6673_`）@HEAD | 基于 `isInvulnerableTo` 的无敌（原版和大多数模组 Boss）。追溯攻击者（直接实体 -> 弹射物 owner），持有能力则 `setReturnValue(false)` |
| **Layer 2** | `PiercingGazeLivingEntityMixin`：`LivingEntity.hurt()`（`m_6469_`）HEAD + RETURN，**栈式递归隔离** | 调了 `super.hurt()` 但被子类强制 return false 的 Boss；hurt 后偷偷 `setHealth` 回血的 Boss。HEAD（`onHurtEnter`）压栈保存外层状态（4 个 ThreadLocal：`IN_PIERCING` / `PIERCING_EVENT_POSTED` / `PIERCING_EFFECTIVE_AMOUNT` / `PIERCING_STACK`）并重置本层；RETURN（`onHurtReturn`）按 A/B/C 三情况补 post 或补 `actuallyHurt`（经 `PiercingGazeLivingEntityAccessor` `m_6475_`），血量没降则 `setAllHealthLikeRaw` 直写扣除，顺带 `InvulClearUtil.clearCustomInvulTimers()`，`finally` 弹栈恢复外层 |
| **Layer 2.5** | 同上类：`@Redirect` 替换 hurt 内部的 `ForgeHooks.onLivingHurt()` 调用 | 在 `LivingHurtEvent` 里 `setAmount()` 压伤害的限伤模组--返回 `max(原始伤害, 事件结果)`，**伤害只能涨不能降**；post 后置 `IN_PIERCING = true` / `PIERCING_EVENT_POSTED = true` / `PIERCING_EFFECTIVE_AMOUNT = effective` 三标记 |
| **Layer 3** | `CombatAbilityHandler.handlePiercingGazeAwakened()`（事件层，`receiveCanceled = true` + `rawAttacker` 回溯） | 觉醒禁无敌帧：目标 `invulnerableTime > 0` 时标记 3 秒（onLivingTick 清零 invulnerableTime） |

**防事件风暴**（v1.2.4）：第三方模组若在 `LivingHurtEvent` 监听器里再调 `target.hurt()`（破敌之眼源），会形成递归事件风暴。`IN_PIERCING` 风暴守卫检测"是否已在外层破敌之眼穿透内"--redirect 手动 post 后置 `IN_PIERCING = true`，递归 hurt 进入时若发现外层 `IN_PIERCING = true` 则跳过手动 post、走原版 `ForgeHooks.onLivingHurt`，阻断递归。redirect 与 `onHurtReturn` 两处各有一个风暴守卫分支。

**三种情况 A/B/C**（`onHurtReturn` 区分拦截位置）：
- **A**：Boss 完全拦截（不调 super、return false）--redirect 未执行，`onHurtReturn` 补 post 一次 `LivingHurtEvent` 再经 Accessor 直调 `actuallyHurt`
- **B**：正常流程（super 走完、`actuallyHurt` 执行、return true）--redirect 已 post 一次，`onHurtReturn` 不再 post
- **C**：中途拦截（走到 `onLivingHurt` 后 return false）--redirect 已 post，`onHurtReturn` 复用 `PIERCING_EFFECTIVE_AMOUNT` 补 `actuallyHurt`，不重复 post

- 跨 Mixin 调用 `actuallyHurt` 走 `PiercingGazeLivingEntityAccessor`（`@Invoker`，refmap 映射 `m_6475_`），不做 `(MixinClass)(Object)` 强转
- 所有层都带客户端跳过（防 BetterCombat 客户端预测导致 `ClientLevel→ServerLevel` 强转崩溃）、内部源防重入、友伤保护

配置：`awaken_piercing_gaze_multiplier`(1.30)

### 死亡抗拒 `death_defy`

**效果**：致死伤害不死，反而回满血 + 清负面 + 60 秒无敌，冷却 5 分钟。觉醒后触发时免费释放一次旅者审判。HUD 显示金色边框和倒计时。

**实现**（三层拦截保证 60 秒无敌"真"无敌）：
1. **事件层**（`LivingDeathEvent` @HIGHEST → `AdventureProgressCapability.onPlayerDeath()`）：冷却检查通过则 `setCanceled(true)`，清 HARMFUL 效果 + `clearFire()`，血量恢复 `max(20, getMaxHealth())`——**双写**：先 `HealthUtil.setHealthDirect`（绕过覆写）再 `player.setHealth`（保证 Forge 监听器和客户端同步）。写入 `deathDefyInvulEnd/CooldownEnd`（Capability NBT 持久化，跨下线）
2. **Mixin 层 1**（`DeathDefyMixin`，`setHealth` @HEAD，priority=1500）：无敌期内拒绝一切降血写入（回血放行），同时过滤 NaN/Infinity
3. **Mixin 层 2**（`tick` @TAIL）：无敌期内每 tick 检查 `getHealthDirect() < 20` 即双写拉回——反制通过 VarHandle 直写 `DataItem.value` 绕过 `setHealth` 的外部手段。钳制目标固定 20 而非 maxHealth，防外部临时改 maxHealth 污染判定
- HUD（`DeathDefyHudOverlay`，`RenderGuiOverlayEvent.Post`）：无敌期内四边金色半透明边框 + 顶部剩余秒数
- 觉醒联动：触发时若启用 `active_skill`，直调 `ActiveSkillHandler.executeJudgment()`，**不消耗冷却不占 GCD**

配置：`death_defy_invul_duration`(1200) / `death_defy_cooldown_duration`(6000)

### 影杀 `shadow_kill`

**效果**：为攻击目标投影一条"影子血量"（紫色 Boss 栏显示，多人各自独立）。每次攻击削减 `4 + 目标总血量 4%`，与目标真实血量无关--**无论 Boss 怎么回血、锁血、免疫，影子血量只减不增，归零即斩杀**。觉醒后斩杀触发 8 格 AOE，削减周围怪物 15% 影子血量并可连锁斩杀。PVP 无效。

**实现**：
- 入口：`LivingHurtEvent` → `CombatAbilityHandler.handleShadowKill()`
- **影子血量存在攻击者（玩家）侧的 `persistentData`**（`AP_ShadowHP_Data` → `{目标UUID: {totalHP, shadowHP, endTime}}`）而非目标侧——Boss 无法通过阶段切换、NBT 清理或任何自身手段逃脱。`totalHP` 在首次攻击时快照，防止后续 maxHealth 变化污染计算
- 条目 5 分钟不攻击自动过期（懒清理 + 每 200 tick 全局清理 + 目标死亡/玩家登出清理，含 BossBar 引用防泄漏）
- BossBar 采用双层 Map（`attackerUUID -> targetUUID -> ServerBossEvent`）：多人同打一 Boss 各有独立 BossBar 显示各自影子血量，互不干扰；玩家登出时按攻击者维度整体移除

**斩杀链**（`executeShadowKill`，`KILLING` Set 防重入）：
1. 清 `hurtTime`/`invulnerableTime` 后 `target.hurt(killSource, target.getMaxHealth() * 10F)`（v1.2.4 防溢出：`Float.MAX_VALUE` 在某些模组的 `amount × ratio` 计算中会溢出为 Infinity/NaN 导致卡死，`maxHealth × 10` 足够秒杀且不溢出）
2. 目标还活着（hurt 被完全拦截）→ `saturationKill()` 饱和式处决：
   - `setAllHealthLikeRaw(target, 0)` 直写血量归零
   - 强制掉落装备 + 反射调 `dropAllDeathLoot()`（`m_6668_`）+ 手动 post `LivingDeathEvent`——**战利品不因非常规死亡而丢失**
   - **五重移除**：`remove(KILLED)` → `remove(DISCARDED)` → 反射 `removeDirect` → 字段直写 `setRemovedFieldDirect` → `setRemovedDirect(CHANGED_DIMENSION)`（部分 Boss 的 Mixin 只拦 KILLED/DISCARDED，不拦"换维度"）
   - 发 `ClientboundRemoveEntitiesPacket` 强制客户端消失
   - **`eradicateFromWorld()`**：反射从 Minecraft 内部 6 个实体追踪结构直接抹除——`EntityLookup.byId`、`EntityLookup.byUuid`、`PersistentEntitySectionManager.knownUuids`、`EntityTickList`（active/passive）、`EntitySection.classInstanceMultiMap`（allInstances + per-class byClass 分列表）、`ServerChunkCache.removeEntity`。全程 `Field.get` + 集合 `remove()`，不调用任何实体方法，**外部 Mixin 无从拦截**
- 觉醒 AOE（`shadowKillAoe`）：斩杀后对半径 8 格内至多 16 个 `Monster` 各削减 `totalHP × 15%` 影子血量，归零者递归进入斩杀链

配置：`shadow_kill_flat_damage`(4) / `shadow_kill_hp_ratio`(0.04) / `shadow_kill_data_expire_ticks`(6000) / `awaken_shadow_kill_aoe_radius`(8.0) / `awaken_shadow_kill_aoe_ratio`(0.15) / `awaken_shadow_kill_aoe_max_targets`(16)

### 真实血量 `true_health`

**效果**：把玩家血量备份到 Capability，任何绕过正常伤害流程的血量篡改（直写数据、假死标记、强制 die、注入负值偏移）都会被检测并自动修复。

**实现**（`TrueHealthMixin`，5 个注入点组成四层防护）：

| 注入点 | SRG | 职责 |
|--------|-----|------|
- 觉醒：防秒杀底线--被外部 setHealth 篡改到 0 以下时强制保留 1HP（正常 hurt 打死不受影响）
| `setHealth()` @RETURN | `m_21153_` | **同步层**：只在合法路径更新备份——`HURT_DEPTH > 0`（hurt 调用链内）或回血时同步；外部直调降血**拒绝同步** |
| `isDeadOrDying()` @HEAD | `m_21224_` | 备份 > 0 时强制返回 false，反假死 |
| `die()` @HEAD | `m_6668_` | 备份 > 0 时取消，拦截不经 hurt/setHealth 直接调 `die()` 的秒杀 |
| `tick()` @HEAD | `m_8119_` | **存活性自检**：每 tick 检查 `isRemoved()`/零血/死亡标记但备份 > 0 → `clearRemovedFlag()` + 修复，最后兜底 |

- 修复动作 `repairHealth()` = `setAllHealthLikeRaw(备份值)` + `clearNegativeFloatDeltas()`（清掉外部注入的负值 Float 偏移条目）
- `IN_ON_GET_HEALTH` ThreadLocal 防修复过程中递归触发 `getHealth`
- "合法降血"的判定依赖拒绝篡改能力维护的 `HURT_DEPTH` 标记（见下）
- 觉醒：防秒杀底线--被外部 setHealth 篡改到 0 以下时强制保留 1HP（正常 hurt 打死不受影响）

配置：`true_health_debug_log`(false)

### 拒绝篡改 `reject_manip`

**效果**：拦截外部模组对玩家血量的非法直接修改（不经过伤害流程的 `setHealth` 降血）和 maxHealth 恶意压低。觉醒后把被拦截的降血量按 30% 反弹给最后攻击者。

**实现**（与真实血量分工：本能力拦"写入"，真实血量修"已写入"）：
- **HURT_DEPTH 标记**（`RejectHealthManipMixin`）：`hurt()`（`m_6469_`）@HEAD 时 ThreadLocal 计数 +1，@RETURN 时 -1。由此区分"hurt 内部的合法 setHealth"与"外部直调 setHealth"。服务端主线程单线程，ThreadLocal 天然隔离调用链，无锁无查表
- **setHealth 拦截**（`m_21153_` @HEAD）：NaN/Infinity 直接 cancel；回血放行；`HURT_DEPTH > 0` 放行；其余（外部直调降血）→ `ci.cancel()`
- **maxHealth 底线**（`RejectHealthManipAttributeMixin`，`AttributeInstance.setBaseValue()`（`m_22100_`）@HEAD）：新值 < 6.0（3 颗心）时先 O(1) 查 `ATTR_OWNER` 映射找所有者，未命中（getAttribute 未被调用的早期路径）再 fallback 遍历在线玩家并补登记。是受保护玩家则 cancel
- 共享的 `ATTR_OWNER` Map 放在 `RejectHealthManipUtil` 工具类（Mixin 类禁止非 private static 成员）；`remove()`（`m_142687_`）@HEAD 注入 `cleanupOnRemoval`，实体卸载时从 `ATTR_OWNER` 移除其映射，防内存泄漏
- 觉醒反弹：`(当前血量 - 目标写入值) × 30%` 以魔法伤害反弹给 `getLastHurtByMob()`

配置：`awaken_reject_manip_reflect_ratio`(0.30)

### 旅者之力 `active_skill`

**效果**：解锁两个主动技能（Y 切换 / G 释放，共享 0.5 秒 GCD）：
- **旅者审判**（30 秒 CD）：对周围 6 格敌对生物造成 `10 + (最大生命 + 当前生命) × 3% × 里程碑数` 的范围伤害。觉醒范围 ×1.5（9 格）
- **旅者庇护**（90 秒 CD）：5 秒无敌 + 清负面 + 灭火。觉醒后庇护期间可以 30% 速度移动

**实现**：

- **审判**（`executeJudgment`）：自定义伤害类型 `adventure_power:judgment`；目标筛选排除玩家/驯服生物/铁傀儡/雪傀儡，仅 `Monster`；带与淬魂相同的**直写补刀**（实际伤害不足 → `setAllHealthLikeRaw` 修正，归零强制 `die()`）
- **庇护**（v1.2.1 补全无敌与移动锁）：
  - 写 Capability 的 `sanctuaryInvulEnd`，过期由 `onPlayerTick` 清零
  - `PlayerStateHandler.onSanctuaryDamage`（`LivingAttackEvent` @HIGHEST）在无敌期内 `event.setCanceled(true)`，但 `DamageTypeTags.BYPASSES_INVULNERABILITY`（`/kill`、虚空等）**照常放行**，防止玩家用庇护规避管理员指令或掉出世界
  - **非觉醒**期间移动速度锁 0（完全定住不能走），**觉醒**才允许 `AWAKEN_SANCTUARY_SPEED`（默认 0.3）倍减速移动；庇护结束时精准恢复移动速度属性基值
  - `event.isCanceled()` 前置检查，避免重复拦截其他能力已取消的事件
- 冷却全部存 Capability（NBT 持久化，重登不重置），释放后 `syncToClient` 驱动 HUD（`ActiveSkillHudOverlay`）显示冷却环
- 觉醒联动：死亡抗拒触发时直调 `executeJudgment()`，绕过 CD 和 GCD（免费审判）

配置：`active_skill_judgment_base_damage`(10.0) / `active_skill_judgment_hp_ratio`(0.03) / `active_skill_judgment_radius`(6.0) / `active_skill_judgment_cooldown`(600) / `active_skill_sanctuary_duration`(100) / `active_skill_sanctuary_cooldown`(1800) / `active_skill_gcd`(10) / `awaken_judgment_range_mult`(1.5) / `awaken_sanctuary_speed`(0.3)
