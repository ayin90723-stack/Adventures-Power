## 总体架构

### 三层实现模型

25 种能力按对抗强度分布在三个实现层，**优先使用高层，仅在高层无法覆盖时才下沉**：

| 层 | 手段 | 特点 | 典型能力 |
|----|------|------|----------|
| **事件层** | Forge 事件（`LivingHurtEvent`、`PlayerTickEvent` 等） | 兼容性最好，其他模组可感知 | 灵巧、大地之力、伤害抗性 |
| **Mixin 层** | 字节码注入原版方法（`hurt`、`setHealth`、`canBeAffected` 等） | 拦截事件系统覆盖不到的路径 | 净魂、不朽装备、死亡抗拒 |
| **反射直写层** | `HealthUtil` 直接读写 `SynchedEntityData` 内部字段、实体追踪结构 | 对抗其他模组的 Mixin/ASM 拦截，"最终手段" | 影杀、真实血量、见既斩兜底 |

设计约定是**事件驱动优先**：能力的业务逻辑尽量放在事件层 Handler 中，Mixin 只做最低层拦截，反射直写只作为被外部模组拦截后的兜底补刀。

### 生效门禁模型

每个能力的效果代码执行前都要通过统一的门禁链：

```
服务端检查（isClientSide 直接 return）
  → isAdventurer() 或 isFullyUnlocked()   佩戴冒险饰品激活的身份
  → isAbilityEnabled("<id>")              玩家可在能力面板（P 键）自由开关
  → （部分能力）isFullyUnlocked()          觉醒专属门禁
```

**注意：以下 5 个能力在代码中硬性要求 `isFullyUnlocked()`，即佩戴「冒险的终点」（全部 10 里程碑达成）后才实际生效**，尽管它们在里程碑 8~10 就已"解锁"：

- 受击坚韧 `resilience`
- 死亡抗拒 `death_defy`
- 真实血量 `true_health`
- 拒绝篡改 `reject_manip`
- 旅者之力 `active_skill`

### 数值成长公式

有成长的能力，其数值随**已解锁里程碑数**（`count`）线性或步进增长，`countAtUnlock` 为该能力所属里程碑的序号（由 `MilestoneRegistry` 从 milestones.json 加载后动态填充）：

```
线性：value = BASE + PER_MILESTONE × (count - countAtUnlock)      // 灵巧、伤害抗性等
步进：value = BASE + STEP × ((count - countAtUnlock) / 2)         // 鸿运当头、休养生息等（每 2 里程碑一档）
固定：value = -1                                                   // 解锁即完整（灵魂绑定、净魂、不朽装备等）
```

### 觉醒机制（第 11 阶段）

全部里程碑达成并持有「冒险的终点」后 `isFullyUnlocked()` 为 true，各 Handler 在门禁后追加觉醒分支：

- **11 个数值强化能力**：数值 × `AWAKEN_MULTIPLIER`（默认 1.3），部分带硬上限钳制（如闪避/减伤 ≤ 95%）
- **14 个机制质变能力**：新增行为（三段跳、AOE 斩杀、免费审判、过量吸血转护盾等），使用各自独立的觉醒配置项

## 共享基础设施

多个能力共用以下工具类，后文不再重复展开。

### HealthUtil — 血量反射工具（`util/HealthUtil.java`)

对抗外部模组篡改血量的核心武器库。原理：原版血量存储在 `SynchedEntityData`（实体同步数据）中，`getHealth()`/`setHealth()` 只是访问器——外部模组常用 Mixin/ASM 拦截访问器，但很难拦截对底层数据结构的直接读写。

| 方法 | 用途 |
|------|------|
| `getHealthDirect()` | 反射拿 `DATA_HEALTH_ID`（SRG `f_20961_`）直接从 `SynchedEntityData` 读真实血量，绕过 `getHealth()` 覆写 |
| `setHealthDirect()` | 同理直接写入，绕过 `setHealth()` 覆写 |
| `setAllHealthLikeDirect()` | 写原版血量 + 通过 `scanCustomHealthKeys()` 扫描实体类继承链上所有 `EntityDataAccessor<Float>` 字段（带缓存），把自定义血量条目一并写入——对付把血存在自定义字段里的 Boss |
| `setAllHealthLikeRaw()` | 三步饱和写入：快照当前血量 → `setAllHealthLikeDirect` → 遍历 `SynchedEntityData.itemsById` 的所有 `DataItem`，对值≈快照值的条目**直接写 `DataItem.value` 字段**（支持 Float/Double/Integer），完全不经过 `data.set()`，绕过 dirty 标记、监听器和一切 Mixin |
| `clearNegativeFloatDeltas()` | 把 DataItem 中所有负值 Float 归零。反制某些 Boss 向玩家注入"负血量偏移量"再 ASM 篡改 `getHealth()` 叠加的手法。**仅供 TrueHealth 防御侧调用，攻击侧不用** |
| `restoreFoodData()` | 反射直写 `FoodData.foodLevel`（`f_38696_`）= 20、`saturationLevel`（`f_38697_`）= 20，绕过 `setFoodLevel()` 拦截 |
| `removeDirect()` / `setRemovedDirect()` / `setRemovedFieldDirect()` | 三种强度的实体移除：反射调 `remove()` → 反射调 private `setRemoved()`（不参与虚分派）→ 直接写 `Entity.removalReason` 字段（`f_146795_`） |
| `clearRemovedFlag()` | 写 `null` 到 `removalReason`，把被外部标记移除的玩家"复活" |
| `eradicateFromWorld()` | 从 Minecraft 内部 6 个实体追踪结构直接抹除实体（详见[影杀](#影杀-shadow_kill)） |

### InvulClearUtil — 自定义无敌清除

`clearCustomInvulTimers(LivingEntity)` 遍历实体类继承链上的所有字段，按命名模式反射清除模组 Boss 的自定义无敌：

- `int` 字段名含 `invul` → 置 0（如 Goety 的 `moddedInvul` / `obsidianInvul`）
- `boolean` 字段名含 `invul` → 置 false
- `boolean` 字段名含 `vulnerable` 且不含 `invul` → 置 true（如 Mowzie's Mobs 钢铁守护者的 `vulnerable`）

### FriendlyFireProtection — 友伤保护

`isOwnerTarget(attacker, target)`：目标是 `TamableAnimal` 且主人是攻击者本人时返回 true。所有"超模"进攻能力（淬魂、影杀、见既斩、审判、嗜血）在生效前都先过这个检查，保证不会误伤自己的狼、猫或其他模组的驯服生物。

### FortuneContext — 跨 Mixin 上下文传递

两个 `ThreadLocal`（`CURRENT_BREAKER` / `AWAKENED`），解决"Mixin 注入点参数里没有玩家引用"的经典问题（详见[鸿运当头](#鸿运当头-fortune_favor)）。

## 里程碑 1 · 初次夜冕

### 灵巧 `agility`

**效果**：受到任何伤害时有概率完全闪避（伤害归零，不触发后续减伤链）。闪避率 10% → 73%（随里程碑成长），觉醒后 ×1.3 封顶 95%。

**实现**：
- 入口：`LivingAttackEvent` → `CombatAbilityHandler.onLivingAttack()`
- 通过门禁后计算闪避率 `(AGILITY_BASE + AGILITY_PER_MILESTONE × 成长数) / 100`，`player.getRandom().nextFloat()` 判定
- 命中闪避则 `event.setCanceled(true)` **在伤害进入减伤/护甲计算前整体取消**，并生成 20 枚 `WITCH` 粒子反馈
- 觉醒：概率 × `AWAKEN_MULTIPLIER`，硬钳制 ≤ 0.95，防止完全无敌

配置：`agility_base`(10) / `agility_per_milestone`(7) / `awaken_multiplier`(1.3)

### 大地之力 `digging_power`

**效果**：挖掘速度 ×1.3 → ×1.75，觉醒后约 ×2.28。

**实现**：
- 入口：`PlayerEvent.BreakSpeed` → `ExplorationAbilityHandler.onBreakSpeed()`
- `event.setNewSpeed(originalSpeed × multiplier)`，纯事件层实现，无 Mixin，兼容性最好

配置：`digging_power_base`(1.3) / `digging_power_per_milestone`(0.05) / `awaken_multiplier`(1.3)

### 恩赐永驻 `perpetual_blessing`

**效果**：正面药水效果持续时间低于保底阈值时自动续期，Buff 永不自然耗尽。玩家可在 Buff 管理面板（O 键）把不想续的效果加入黑名单。

**实现**：
- 入口：`PlayerTickEvent`(END) → `AdventureProgressCapability.onPlayerTick()`，用 `lastBuffCheck` Map 节流为每 60 tick 检查一次
- `extendBeneficialEffects()`：遍历 `player.getActiveEffects()`，只处理 `MobEffectCategory.BENEFICIAL`；效果剩余时间 < `阈值 = BUFF_MIN_DURATION + BUFF_EXTEND_AMOUNT` 时用新 `MobEffectInstance` 覆盖（保留 amplifier/ambient/visible 参数）
- 黑名单存在 `player.getPersistentData()` 的 `MME_BuffBlacklist` 键下，跨死亡持久
- 觉醒：延长量 × `AWAKEN_MULTIPLIER`

配置：`buff_min_duration`(400) / `buff_extend_amount` / `awaken_multiplier`(1.3)

## 里程碑 2 · 初尝败绩

### 虚空踏步 `void_step`

**效果**：空中可再次跳跃（二段跳），保留疾跑冲量、清空摔落距离。觉醒后升级为三段跳。

**实现**（客户端预测 + 服务端权威的完整链路）：
1. **客户端**（`JumpInputHandler`，监听 `ClientTickEvent`）：按住空格检测（非边沿触发），维护 10 tick 起跳冷却（对齐原版 `noJumpDelay`）；条件满足时本地立即 `jumpFromGround()` 给即时手感，同时发送 `DoubleJumpPacket`
2. **服务端**（`DoubleJumpHandler.handleDoubleJump()`，网络包处理）：重新校验门禁（v1.1.8 加固：客户端不再无条件跳跃），`canDoubleJump()` 检查不骑乘/不在水中/不在飞行/不在地面，`JUMPS_USED` Map 按 UUID 计数
3. 跳跃力：`0.42 × 方块 JumpFactor + 跳跃提升加成`，整体 ×1.2；疾跑时沿视线方向附加 0.2 水平冲量；`fallDistance = 0`；`ClientboundSetEntityMotionPacket` 回同步
4. 生命周期：落地 / 登出 / 重生（`Clone`）/ 换维度时清理 `JUMPS_USED`，防止计数残留
- 觉醒：`getMaxAirJumps()` 返回 `AWAKEN_VOID_STEP_JUMPS - 1`（默认 3-1=2 次空中跳），即**三段跳**

配置：`awaken_void_step_jumps`(3)

### 休养生息 `rapid_recovery`

**效果**：脱离战斗 5 秒后，每 3 秒自动回血并回满饱食度。回量随里程碑成长，觉醒每周期额外 +2 HP。

**实现**（v1.1.9 起为直写方案）：
- 受伤记录：`RecoveryHandler.onLivingHurt()` 把每次受击时间写入 `lastHurtTimestamps[UUID]`
- 恢复判定：`PlayerTickEvent`(END) 每 60 tick 检查一次，`当前时间 - 上次受伤 ≥ RAPID_RECOVERY_DELAY_TICKS`（100 tick）判定为脱战
- **直写回血**：`HealthUtil.setAllHealthLikeRaw(player, min(maxHP, 当前 + (amplifier+1)))`。**不使用再生药水效果**，避免被外部模组的 `MobEffectEvent` 拦截或 `removeAllEffects` 清除
- **直写饱食度**：`HealthUtil.restoreFoodData()` 反射写 `FoodData` 字段为 20/20
- 登录时 `putIfAbsent` 初始化受伤时间，防止刚上线被误判为"已脱战"立刻回血；登出清理两个 Map 防内存泄漏
- 觉醒：`amplifier += AWAKEN_RAPID_RECOVERY_BONUS`(2)

配置：`rapid_recovery_amplifier_base`(0) / `rapid_recovery_amplifier_step`(1) / `rapid_recovery_delay_ticks`(100) / `rapid_recovery_check_interval`(60) / `awaken_rapid_recovery_bonus`(2)

## 里程碑 3 · 初次交易

### 灵魂绑定 `soul_bind`

**效果**：死亡时保留全部药水效果；觉醒后经验也完全不掉落。

**实现**（利用 persistentData 跨越死亡）：
- **死亡时**（`LivingDeathEvent` @HIGHEST）：把 `getActiveEffects()` 全部序列化为 `CompoundTag` 存入 `player.getPersistentData()`（键 `AP_SoulBind_Buffs`），`totalExperience` 存入 `AP_SoulBind_Exp`。选 persistentData 是因为 Capability 在死亡时会被 `invalidateCaps`
- **觉醒时**额外把 `experienceLevel/experienceProgress/totalExperience` 清零——经验已备份，清零后死亡就不会喷出经验球
- **重生时**（`PlayerEvent.Clone`，`isWasDeath()` 分支）：从 `event.getOriginal()` 的 persistentData 读回，逐个 `MobEffectInstance.load()` + `addEffect()`，`giveExperiencePoints()` 还原经验，随后删除两个键防重复恢复
- 非死亡 Clone（如末地传送门）走 `transferSoulBindData()` 整体转移，防跨维度丢数据

配置：无（解锁即完整）

### 不动如山 `knockback_resist`

**效果**：击退抗性 30% → 79%，觉醒 ×1.3（硬上限 100%）。

**实现**：
- 入口：`KnockbackResistHandler.onPlayerTick()`(END) 每 tick 对账
- **直接写属性基值**：`Attributes.KNOCKBACK_RESISTANCE` 的 `setBaseValue(percent/100)`，让原版击退计算自行消化，与其他模组的击退修改天然兼容（不用 AttributeModifier 是因为只动基值、无 UUID 冲突问题）
- 对账逻辑：能力开而属性不匹配 → 写入；能力关而有值 → 归零；`Clone` 重写、登出归零防残留

配置：`knockback_resist_base`(30) / `knockback_resist_per_milestone`(7) / `knockback_resist_hard_cap`(100.0) / `awaken_multiplier`(1.3)

## 里程碑 4 · 初探地底

### 伤害抗性 `damage_resist`

**效果**：受到的所有伤害减免 10% → 40%，觉醒 ×1.3 封顶 95%。

**实现**：
- 入口：`LivingHurtEvent` → `CombatAbilityHandler.handleDamageResist()`
- `event.setAmount(amount × (1 - ratio))`，最轻量的事件层减伤

配置：`damage_resist_base`(10) / `damage_resist_per_milestone`(5) / `awaken_multiplier`(1.3)

### 无形之手 `extended_reach`

**效果**：方块交互距离 +1.0 → +2.2 格，觉醒 ×1.3。不影响攻击距离。

**实现**：
- 入口：`ExplorationAbilityHandler.onPlayerTick()`(END) 每 tick 对账
- 写 Forge 专有属性 `ForgeMod.BLOCK_REACH`（只管方块，不碰实体攻击距离）：`expected = 默认值 + bonus`，误差 > 0.001 才写入（防抖）
- `Clone` 恢复、登出重置为默认值

配置：`extended_reach_base`(1.0) / `extended_reach_per_milestone`(0.2) / `awaken_multiplier`(1.3)

## 里程碑 5 · 初次附魔

### 不朽装备 `undying_gear`

**效果**：所有装备耐久永不消耗。觉醒后每件护甲 +1 护甲值、主手武器 +15% 伤害（独立配置）。

**实现**（`LegacyHurtAndBreakMixin`，双注入点缺一不可）：
- `ItemStack.hurtAndBreak()`（SRG `m_41622_`）@HEAD：原版耐久扣除入口，`ci.cancel()` + `setDamageValue(0)` 回满
- `ItemStack.hurt()`（SRG `m_220157_`）@HEAD：**Forge 1.20.1 新增的独立耐久扣除方法，不经过 hurtAndBreak**，部分模组直接调它。`cir.setReturnValue(false)`（"未损坏"）+ `setDamageValue(0)`
- 门禁走静态方法 `AdventurePower.hasUndyingGear(player)`（Mixin 类中禁止非 private static，门禁逻辑放主类）
- 觉醒加成不在此 Mixin，由属性 Handler 单独处理

配置：`awaken_undying_armor_bonus` / `awaken_undying_weapon_bonus`

### 鸿运当头 `fortune_favor`

**效果**：时运与抢夺等级 +1 → +3（每 2 里程碑一档），在工具已有附魔上叠加。觉醒额外 +2。

**实现**（时运与抢夺路径完全不同）：
- **抢夺**：`LootingLevelEvent` → `FortuneFavorHandler.onLootingLevel()`，`event.setLootingLevel(原值 + bonus)`——Forge 有现成事件，纯事件层
- **时运**：原版没有对应事件，需要 Mixin + 上下文传递：
  1. `BlockEvent.BreakEvent` 时：`FortuneContext.clear()`（先清防残留）→ `setBreaker(player)`，觉醒时 `setAwakened(true)`
  2. `FortuneFavorMixin` 注入 `EnchantmentHelper.getItemEnchantmentLevel()`（SRG `m_44843_`）@RETURN：查询的是 `BLOCK_FORTUNE` 时读 `FortuneContext.getBreaker()`，算出 bonus 后 `cir.setReturnValue(原值 + bonus)`
- **为什么要 ThreadLocal 上下文**：`getItemEnchantmentLevel(Enchantment, ItemStack)` 的参数里没有玩家——`ItemStack` 不持有主人引用，无法从注入点推导"谁在挖"。`BreakEvent` 与附魔查询在同一调用链上，用 `ThreadLocal` 桥接

配置：`fortune_favor_bonus_base`(1) / `fortune_favor_bonus_step`(1) / `awaken_fortune_favor_bonus`(2)

## 里程碑 6 · 炽热之门

### 环境免疫 `env_immunity`

**效果**：免疫火焰、冻结、雷击、摔落、溺水、仙人掌、凋零等环境伤害。觉醒后免疫**一切无实体来源的伤害**。

**实现**：
- 入口：`LivingAttackEvent` → `PlayerStateHandler.onEnvDamage()`，在伤害进入结算前取消
- 判定顺序：
  1. 觉醒分支：`source.getEntity() == null && source.getDirectEntity() == null`（完全无实体来源）→ 直接取消，不限类型
  2. 非觉醒：有实体来源的一律放行（火焰弹的火伤算战斗伤害，不免疫）
  3. 标签匹配：`DamageTypeTags.IS_FIRE / IS_FREEZING / IS_LIGHTNING / IS_FALL / IS_DROWNING`
  4. 字符串兜底：标签没覆盖的类型（仙人掌、甜浆果、钟乳石、窒息、岩浆块、饥饿、凋零、干涸）按 `source.getMsgId()` 比对
- 辅助：`onPlayerTick` 每 tick `player.clearFire()`，对付直接 `setRemainingFireTicks` 的模组

配置：无

### 嗜血 `lifesteal`

### 真实血量 `true_health` ⚠️ 觉醒专属

**效果**：把玩家血量备份到 Capability，任何绕过正常伤害流程的血量篡改（直写数据、假死标记、强制 die、注入负值偏移）都会被检测并自动修复。

**实现**（`TrueHealthMixin`，5 个注入点组成四层防护）：

| 注入点 | SRG | 职责 |
|--------|-----|------|
| `getHealth()` @HEAD | `m_21223_` | **读取层**：返回备份值；检测 NaN/Infinity、假死（DataItem≤0 但备份>0）、非法降血（DataItem < 备份 - ε）→ 触发修复 |
| `setHealth()` @RETURN | `m_21153_` | **同步层**：只在合法路径更新备份——`HURT_DEPTH > 0`（hurt 调用链内）或回血时同步；外部直调降血**拒绝同步** |
| `isDeadOrDying()` @HEAD | `m_21224_` | 备份 > 0 时强制返回 false，反假死 |
| `die()` @HEAD | `m_6668_` | 备份 > 0 时取消，拦截不经 hurt/setHealth 直接调 `die()` 的秒杀 |
| `tick()` @HEAD | `m_8119_` | **存活性自检**：每 tick 检查 `isRemoved()`/零血/死亡标记但备份 > 0 → `clearRemovedFlag()` + 修复，最后兜底 |

- 修复动作 `repairHealth()` = `setAllHealthLikeRaw(备份值)` + `clearNegativeFloatDeltas()`（清掉外部注入的负值 Float 偏移条目）
- `IN_ON_GET_HEALTH` ThreadLocal 防修复过程中递归触发 `getHealth`
- "合法降血"的判定依赖拒绝篡改能力维护的 `HURT_DEPTH` 标记（见下）
- 觉醒：检测 ε ×2（0.001→0.002），即修复判定更宽容、修复覆盖更快的写入频率

### 拒绝篡改 `reject_manip` ⚠️ 觉醒专属

**效果**：拦截外部模组对玩家血量的非法直接修改（不经过伤害流程的 `setHealth` 降血）和 maxHealth 恶意压低。觉醒后把被拦截的降血量按 30% 反弹给最后攻击者。

**实现**（与真实血量分工：本能力拦"写入"，真实血量修"已写入"）：
- **HURT_DEPTH 标记**（`RejectHealthManipMixin`）：`hurt()`（`m_6469_`）@HEAD 时 ThreadLocal 计数 +1，@RETURN 时 -1。由此区分"hurt 内部的合法 setHealth"与"外部直调 setHealth"。服务端主线程单线程，ThreadLocal 天然隔离调用链，无锁无查表
- **setHealth 拦截**（`m_21153_` @HEAD）：NaN/Infinity 直接 cancel；回血放行；`HURT_DEPTH > 0` 放行；其余（外部直调降血）→ `ci.cancel()`
- **maxHealth 底线**（`RejectHealthManipAttributeMixin`，`AttributeInstance.setBaseValue()`（`m_22100_`）@HEAD）：新值 < 6.0（3 颗心）时遍历在线玩家反向匹配属性实例所有者，是受保护玩家则 cancel。不用 `getAttribute()` 注入时记录的 `ATTR_OWNER` 映射做主判定，因为外部可直接访问 `AttributeMap` 绕过 `getAttribute()`
- 共享的 `ATTR_OWNER` Map 放在 `RejectHealthManipUtil` 工具类（Mixin 类禁止非 private static 成员）
- 觉醒反弹：`(当前血量 - 目标写入值) × 30%` 以魔法伤害反弹给 `getLastHurtByMob()`

**效果**：解锁两个主动技能（Y 切换 / G 释放，共享 0.5 秒 GCD）：
- **旅者审判**（30 秒 CD）：对周围 6 格敌对生物造成 `10 + (最大生命 + 当前生命) × 3% × 里程碑数` 的范围伤害。觉醒范围 ×1.5（9 格）
- **旅者庇护**（90 秒 CD）：5 秒无敌 + 清负面 + 灭火。觉醒后庇护期间可以 30% 速度移动

**实现**：
- 链路：客户端 `InputHandler`（G 键，本地预检 GCD/CD 减少无效包）→ `ActiveSkillPacket` → 服务端 `ActiveSkillHandler.handleSkillRelease()` **重新完整校验**（skillIndex 范围、`isFullyUnlocked`、能力启用、GCD、各技能 CD——v1.1.8 加固的防御性自门禁）
- **审判**（`executeJudgment`）：自定义伤害类型 `adventure_power:judgment`；目标筛选排除玩家/驯服生物/铁傀儡/雪傀儡，仅 `Monster`；带与淬魂相同的**直写补刀**（实际伤害不足 → `setAllHealthLikeRaw` 修正，归零强制 `die()`）
- **庇护**：写 Capability 的 `sanctuaryInvulEnd`，由伤害处理链读取 `isSanctuaryInvulnerable()` 判定免伤；过期由 `onPlayerTick` 清零
- 冷却全部存 Capability（NBT 持久化，重登不重置），释放后 `syncToClient` 驱动 HUD（`ActiveSkillHudOverlay`）显示冷却环
- 觉醒联动：死亡抗拒触发时直调 `executeJudgment()`，绕过 CD 和 GCD（免费审判）

配置：`active_skill_judgment_base_damage`(10.0) / `active_skill_judgment_hp_ratio`(0.03) / `active_skill_judgment_radius`(6.0) / `active_skill_judgment_cooldown`(600) / `active_skill_sanctuary_duration`(100) / `active_skill_sanctuary_cooldown`(1800) / `active_skill_gcd`(10) / `awaken_judgment_range_mult`(1.5) / `awaken_sanctuary_speed`(0.3)
