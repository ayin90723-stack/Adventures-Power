# 18 种能力实现审查

## 实现模式分类

| 模式 | 能力数 | 说明 |
|------|--------|------|
| Forge 事件拦截 | 11 | 通过 `@SubscribeEvent` 在关键事件点注入逻辑 |
| Mixin 底层注入 | 5 | 无法通过事件实现的，走字节码注入 |
| Tick 周期维护 | 3 | 需要每帧维持状态的能力 |

---

## 逐一审查

### 1. 灵巧 (Agility) — 概率闪避

**实现位置**：`CombatAbilityHandler.onLivingAttack()`

```
流程：LivingAttackEvent → 检查 agility 启用 → 取 value(milestones)/100 得概率
     → random < chance → event.setCanceled(true) + 粒子特效
```

**评价**：✅ 合理

- `LivingAttackEvent` 是伤害计算前的最后一道拦截点，在此取消可以阻止后续所有伤害处理（护甲/附魔/无敌帧），是最干净的闪避实现位置
- 使用 Capability 里程碑数动态计算概率，里程碑越多越能闪
- 粒子反馈让玩家知道闪避生效了

---

### 2. 恩赐永驻 (Perpetual Blessing) — Buff 保底续期

**实现位置**：`AdventureProgressCapability.onPlayerTick()`

```
流程：每 3 秒检查活跃效果 → 正面效果持续时间 < 阈值 → 刷新到阈值
```

**评价**：✅ 合理

- 每 3 秒（60 tick）而非每 tick 检查，性能友好
- 支持黑名单排除（BuffManagementScreen 管理），玩家可自定义不想永续的 Buff
- `ForgeRegistries.MOB_EFFECTS.getKey()` 作为唯一标识，跨模组兼容

---

### 3. 虚空踏步 (Void Step) — 二段跳

**实现位置**：`JumpInputHandler`（客户端监听空格）→ `NetworkHandler.DoubleJumpPacket` → `DoubleJumpHandler.handleDoubleJump()`（服务端执行）

```
流程：客户端检测空中按空格 → 发包 → 服务端检查能力启用/冷却/状态
     → 计算跳跃力（基础0.42 × 方块系数 + 跳跃提升加成）→ setDeltaMovement
```

**评价**：✅ 合理

- 客户端只发信号，服务端做所有逻辑（防作弊）
- 复用原版跳跃公式（`0.42 × getJumpFactor + 0.1 × 跳跃等级`），跳跃提升药水自然作用于二段跳
- 1.2 倍力度的增幅适度，不会过于破坏平衡
- 有 20 tick 冷却防止连续触发

---

### 4. 灵魂绑定 (SoulBind) — 死亡保 Buff + 经验

**实现位置**：`PlayerStateHandler.onPlayerDeath()` + `onPlayerClone()`

```
死亡前：LivingDeathEvent(HIGHEST) → 序列化 activeEffects + totalExperience → persistentData
重生后：PlayerEvent.Clone(isWasDeath=true) → 从旧玩家 persistentData 读回 → addEffect + giveExperiencePoints
```

**评价**：✅ 合理，有一个小隐患

- `LivingDeathEvent` 优先级 `HIGHEST` 确保在其他 handler 修改死亡状态前保存数据
- 使用 `MobEffectInstance.save()` / `MobEffectInstance.load()` 标准序列化，兼容所有原版和模组效果
- 恢复时调用 `player.giveExperiencePoints()` 而非直接写字段，走正常经验管线（触发经验条动画等）

**⚠️ 隐患**：`DeathDefy`（死亡抗拒）也会监听 `LivingDeathEvent(HIGHEST)` 并取消死亡。如果两者同时触发，灵魂绑定会先保存 Buff，然后死亡抗拒取消死亡——但 Buff 不会被恢复（因为 `onPlayerClone` 只处理 `isWasDeath()`）。**然而这不是 bug**：死亡被取消意味着玩家没死，Buff 还在身上，不需要恢复。

---

### 5. 伤害抗性 (DamageResist) — 全伤害减免

**实现位置**：`CombatAbilityHandler.handleDamageResist()`（LivingHurtEvent）

```
流程：LivingHurtEvent → 检查 damage_resist 启用 → value(milestones)/100 得比例
     → event.setAmount(amount * (1 - ratio))
```

**评价**：✅ 合理

- `LivingHurtEvent` 在护甲/附魔减伤之后，调整的是最终伤害值
- `setAmount(amount * (1 - ratio))` 简单直接，与所有其他减伤叠加（乘法）

---

### 6. 不朽装备 (UndyingGear) — 装备免耐久

**实现位置**：`LegacyHurtAndBreakMixin.onHurtAndBreak()`

```
注入点：ItemStack.hurtAndBreak() HEAD
逻辑：if (entity instanceof Player && hasUndyingGear(player)) → ci.cancel() + setDamageValue(0)
```

**评价**：✅ 合理

- `hurtAndBreak()` 是所有耐久消耗的唯一入口（挖掘/攻击/受伤/铁砧都走这里）
- 注入 HEAD + cancel 是最干净的拦截方式——耐久根本没被扣，连后续的碎裂检查都不会触发
- `setDamageValue(0)` 额外保证：如果物品以某种方式有了耐久损伤（如铁砧直接写 NBT），也会被清零

---

### 7. 环境免疫 (Env Immunity) — 免疫火焰/岩浆

**实现位置**：`AdventureProgressCapability.onPlayerTick()`

```
每 tick：player.clearFire()
```

**评价**：✅ 合理但覆盖面有限

- `clearFire()` 会清除火焰 tick 计数和火焰 overlay。由于岩浆伤害依赖 `isInLava()` + 火焰 tick，清除火焰 tick 也间断了岩浆伤害链
- **不足之处**：对仙人掌、窒息、虚空等非火焰环境伤害无效。但设计文档写明"免疫火焰、岩浆等环境伤害"，当前实现匹配设计意图
- 如果需要全面环境免疫，可扩展为检查 `DamageSource` 类型

---

### 8. 禁疗之触 (HealingBlock) — 攻击施加禁疗

**实现位置**：`CombatAbilityHandler.handleHealingBlock()`（LivingHurtEvent）

```
流程：攻击者 LivingHurtEvent → 检查 healing_block 启用 → 计算持续时间(秒×20=tick)
     → UndyingSlashEffect.apply(target, durationTicks)
```

`UndyingSlashEffect` 内部实现：
```
apply(): NBT 写入到期时间 + 记录施加时血量
isActive(): 检查 NBT 到期时间 → 超时自动清理

效果层（MobEffect + 事件拦截）：
- LivingHealEvent → cancel（禁止回血）
- setHealth() 被 UndyingSlashMixin 拦截 → 非 hurt 路径回血钳制
- LivingUseTotemEvent → cancel（反复活）
- tick 末 → HealthUtil 钳制血量到追踪的低点
```

**评价**：✅ 合理

- NBT 持久化标记 + MobEffect 视觉指示器分离——NBT 是唯一权威数据源，MobEffect 只是给玩家看的红色粒子
- 禁疗由三层拦截组成（heal 事件 + setHealth mixin + tick 钳制），每一层处理不同的回血路径
- 攻击时只施加标记，不立即造成伤害——禁疗是"软控"而非"直伤"

**⚠️ 时序注意**：`handleSoulQuench` 的禁疗 ×1.5 检查在 `handleHealingBlock` 之前执行，所以本击不会吃到自身施加的禁疗加成。这是正确的——必须先有标记再增伤。

---

### 9. 受击坚韧 (Resilience) — 受击叠层减伤

**实现位置**：
- 叠层+减伤：`PlayerStateHandler.onLivingHurt()` (LOW 优先级)
- 超时归零：`AdventureProgressCapability.onPlayerTick()`

```
受伤时(LOW)：基于已有层数减伤(每层5%) → 层数+1(上限由value决定) → 更新lastHurtTime
每 tick：lastHurtTime 超过阈值 → 层数归零
```

**评价**：✅ 合理

- `LOW` 优先级确保在淬魂等增伤之后执行，不影响伤害加成计算
- 先减伤再叠层：已叠的层数对本击生效，合理——如果先叠层再减伤，第一击就完全没有减伤
- 5 秒无受伤重置避免了"战斗中突然失效"的不公平感，也防止玩家被打后跑路等重置

---

### 10. 净魂 (PurifiedSoul) — 免疫负面效果

**实现位置**：`PlayerStateHandler` — 双重防线

```
第一道防线：MobEffectEvent.Applicable → HARMFUL 类效果 → event.setCanceled(true)
第二道防线（兜底）：PlayerTickEvent → 遍历 activeEffects → 移除 HARMFUL 类
```

**评价**：✅ 合理

- 双重防线设计是必要的：`Applicable` 拦截正常施加路径，tick 清除覆盖绕过 `Applicable` 的路径（如直接 `forceAddEffect`、NBT 写入等）
- tick 中每帧全部清除性能开销很小（通常 0-3 个负面效果，`removeEffect` 是 O(1)）
- 不区分负面效果来源（原版/模组/环境），全部一视同仁

---

### 11. 翱翔 (Soar) — 创造飞行

**实现位置**：`PlayerStateHandler.onPlayerTick()`

```
每 tick：mayfly = true（维持，因为死亡/维度切换会重置）
关闭时：回收飞行，但不干扰创造模式（检查 !instabuild）
```

**评价**：✅ 合理

- 每 tick 设置 `mayfly = true` 而非一次性：死亡和维度切换会重置 `PlayerAbilities`，需要持续维持
- 不自动开启 `flying`：让玩家自己双击跳跃开始飞行，体验更好
- 关闭时检查 `instabuild`：不会误收回创造模式玩家的飞行能力

### 12. 淬魂之力 (SoulQuench) — 攻击附加百分比伤害

**实现位置**：`CombatAbilityHandler.handleSoulQuench()`（LivingHurtEvent）

```
公式：额外伤害 = flatDamage(2→4)
                + target.maxHealth × hpRatio(1%→2%)
                + target.currentHealth × hpRatio(1%→2%)
禁疗标记目标：×1.5
最终：event.setAmount(amount + bonus)
```

**评价**：✅ 合理

- 三部分伤害各有定位：固伤保证对低血量目标有效，maxHP% 对 Boss 特攻，currentHP% 对满血目标先手优势
- 通过 `event.setAmount()` 追加伤害，走正常伤害管线（会被目标的护甲/抗性再次减伤），非真实伤害——这意味着淬魂伤害不是"穿透"，而是"额外攻击力"
- 禁疗 ×1.5 联动让两个能力组合使用时效果更强，但各自独立也能用

**🤔 可讨论**：如果用 `HealthUtil.setAllHealthLikeRaw()` 直写血量绕过护甲，淬魂就变成了真实伤害。当前设计是"额外攻击力"而非"真实伤害"——哪个更符合你的预期？

### 13. 破敌之眼 (PiercingGaze) — 穿透无敌

**实现位置**：3 个 Mixin，4 层穿透架构

```
Layer 0: SeeAndSlashPlayerAttackMixin — Player.attack() 内 target.hurt() 调用前
         → InvulClearUtil 清除目标自定义无敌字段

Layer 1: SeeAndSlashMixin — Entity.isInvulnerableTo() HEAD
         → 持有者直接返回 false

Layer 2: SeeAndSlashLivingEntityMixin — LivingEntity.hurt() RETURN
         → hurt() 返回 false 时调用 actuallyHurt（兜底穿透）

Layer 2.5: 同上 Mixin — ForgeHooks.onLivingHurt() Redirect
           → 绕过 Boss 限伤，伤害只能涨不能降
```

**评价**：✅ 这是整个模组最复杂的系统，每一层解决一类"无敌"：

- Layer 1 处理基于 `isInvulnerableTo()` 的原版无敌（伤害冷却、创造模式等）
- Layer 2 处理 Boss 重写 `hurt()` 返回 false 的自定义无敌
- Layer 2.5 处理 Boss 通过 `LivingHurtEvent` 把伤害钳制为 0 的限伤机制
- Layer 0 处理 Boss 完全重写 `hurt()` 连 `super.hurt()` 都不调用的极端情况

配合 `InvulClearUtil` 反射清除和 `HealthUtil` 直写血量，四层形成了完整的无敌穿透链。

### 14. 死亡抗拒 (Death Defy) — 冷却制免死

**实现位置**：
- 触发：`AdventureProgressCapability.onPlayerDeath()` (HIGHEST)
- 无敌保护：`DeathDefyMixin`（setHealth HEAD + tick TAIL）

```
触发：LivingDeathEvent(HIGHEST) → 检查冷却 → cancel 死亡
     → 清除有害效果 + 灭火 → 固定回满 20HP（不用 getMaxHealth 防止被降上限）
     → 写入无敌/冷却截止时间

无敌期保护：
- setHealth() HEAD: 无敌期间拒绝降血
- tick TAIL: 每 tick 反射写入强制恢复（反制 DataItem 篡改）
```

**评价**：✅ 合理

- `HIGHEST` 优先级确保在其他 handler 之前取消死亡
- 回血用固定值 20 而非 `getMaxHealth()`：某些 mod（如亚波伦结界）会临时把 maxHealth 降为 1
- 双层 Mixin 保护：方法层 + 字段层，即使 setHealth 被绕过，tick 末仍会强制恢复
- HUD 金色边框 + 倒计时提供视觉反馈

### 15. 影杀 (ShadowKill) — 影子血量斩杀

**实现位置**：`CombatAbilityHandler.handleShadowKill()`（LivingHurtEvent）

```
流程：
1. 首次攻击：shadowHP = target.maxHealth（初始化影子血量）
2. 每次攻击：shadowHP -= (flatDamage + maxHealth × hpRatio)
3. shadowHP ≤ 0 → 斩杀：
   - HealthUtil.setAllHealthLikeRaw(target, 0)  // 直写血量
   - target.invulnerableTime = 0                 // 清无敌帧
   - target.hurtDuration = 0                     // 清受伤动画
   - target.die(source)                          // 强制死亡
4. 对玩家无效
```

**评价**：✅ 合理

- 影子血量独立于真实血量，不受目标护甲/抗性/无敌帧影响——这就是"独立斩杀"的核心优势
- 影子血量归零时使用 `HealthUtil.setAllHealthLikeRaw(0)` 而非 `setHealth(0)` 或 `hurt(∞)`：绕过一切血量保护（如 Boss 重写的 setHealth）
- NBT 存储影子血量在 `persistentData` 而非 Capability：影子血量属于"目标身上的标记"，不属于玩家能力数据
- 影子血量耗尽后自动清理 NBT（`remove`），避免内存泄漏

**🤔 可讨论**：当前每次攻击的扣除量 = `4 + maxHealth × 4%`。对于末影龙（200HP）= 4 + 8 = 12，需要约 17 次攻击才能触发斩杀。这个节奏是否合适？

### 16. 真实血量 (TrueHealth) — 备份修复血量

**实现位置**：`TrueHealthMixin`

```
getHealth() HEAD:
  backup > 0 && backup ≠ DataItem 中的值 → 自动修复 DataItem
  isDeadOrDying() 但 backup > 0 → 取消假死

setHealth() RETURN:
  合法 hurt 路径 → 同步 backup = 新血量

die() HEAD:
  backup > 0 → 取消死亡
```

**评价**：✅ 合理

- 备份血量存储在 Capability 上，不受 `SynchedEntityData.set()` 篡改
- `getHealth()` 每次读取都检查一致性并自动修复——对上层代码透明
- 假死检测：如果外部 mod 把 DataItem 设为 ≤ 0 但备份血量还 > 0，强制返回存活状态

### 17. 拒绝篡改 (RejectManip) — 拦截非法血量修改

**实现位置**：
- `RejectHealthManipMixin`：HURT_DEPTH 标记系统
- `RejectHealthManipAttributeMixin`：maxHealth 底线保护

```
HURT_DEPTH 标记：
- hurt() HEAD → HURT_DEPTH = true
- hurt() RETURN → HURT_DEPTH = false
- setHealth() HEAD → HURT_DEPTH == false 时拒绝（非 hurt 路径）

maxHealth 保护：
- AttributeInstance.setBaseValue() HEAD → maxHealth < 6 + 所有者在线上 → cancel
```

**评价**：✅ 合理

- `HURT_DEPTH` 标记区分"合法的 hurt 路径"和"外部 mod 直接调 setHealth()"——只有前者被允许
- `RejectHealthManipUtil.ATTR_OWNER` 维护 AttributeInstance → LivingEntity 映射，用于在 AttributeMixin 中反查所有者
- maxHealth < 6 的阈值是一个合理的底线——正常游戏不会把血量降到这么低

### 18. 旅者之力 (ActiveSkill) — 主动技能

**实现位置**：
- 入口：`ActiveSkillHandler.handleSkillRelease()`（服务端）
- 客户端按键 → NetworkHandler → handleSkillRelease

```
handleJudgment:
  公式 = baseDamage(10) + milestones × hpRatio(3%) × (maxHealth + currentHealth)
  遍历 AOE 范围内敌对生物 → hurt() 伤害 → 清零无敌帧
  兜底：hurt() 被拦截 → HealthUtil 补扣 / 直接 die()

handleSanctuary:
  写入无敌截止时间 + 冷却时间 + GCD
  清除有害效果 + 灭火
```

**评价**：✅ 合理

- GCD（全局冷却 0.5s）防止判断和庇护同时释放
- 审判伤害包含 maxHP + currentHP 两部分，对 Boss 和满血小怪都有威胁
- 兜底机制：如果 hurt() 被外部 mod 拦截（配合破敌之眼场景），用 HealthUtil 直写血量
- 伙伴保护（`FriendlyFireProtection.isOwnerTarget`）确保不误伤自己的驯服生物
